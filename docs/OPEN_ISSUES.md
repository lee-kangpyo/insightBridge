# Open Issues

## 1. 데이터 가공 방식

### 문제
LLM이 SQL을 생성해서 데이터를 가져온 후, 가공 처리해야 할 때 어떻게 할 것인가?

예시: "월별 매출의 증가율을 보여줘"
- SQL로 월별 매출 데이터 조회 가능
- 증가율 계산은 SQL로도 가능하지만 복잡

### 옵션 A: 백엔드에서 pandas 가공
```
LLM → SQL 생성 → SQL 실행 → 백엔드에서 pandas 가공 → 프론트
```
- 장점: 빠름, 저렴
- 단점: 복잡한 가공은 LLM 2차 호출 필요

### 옵션 B: LLM이 pandas 코드 생성
```
LLM → SQL 생성 → SQL 실행 → pandas DataFrame → LLM에 전송
→ LLM이 pandas 코드 생성 → 백엔드에서 실행 → 프론트
```
- 장점: 복잡한 가공 용이
- 단점: 느림, 비용 ↑, 보안 위험

### 결론
**미결정** - 구현 후 성능/비용 테스트 후 결정

## 2. LLM 생성 pandas 코드 보안

### 문제
옵션 B 채택 시, LLM이 생성한 pandas 코드를 백엔드에서 실행해야 함

### 위험
- `os.system()`, `subprocess` 등 시스템 명령어 실행 가능
- 무한 루프 가능
- 메모리 과소비 가능

### 가능한 완화 방안
1. **화이트리스트 방식**
   ```python
   ALLOWED_OPERATIONS = ['groupby', 'sum', 'mean', 'pivot', 'merge', ...]
   # 파싱해서 허용된 operation만 실행
   ```

2. **샌드박스 환경**
   - Docker 컨테이너에서 실행
   - 리소스 제한 (시간, 메모리)

3. **AST 파싱 + 검증**
   - `pandas` 모듈의 메서드만 허용
   - `pd.DataFrame` 리턴必须是

### 결론
**미결정** - 옵션 A/B 결정 후 진행
